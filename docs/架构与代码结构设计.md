## 2. 执行引擎架构

### 2.1 执行流程

```text
触发执行（手动/Timer/Webhook/API）
  │
  ▼
校验（剧本是否存在、状态是否可执行、并发限制）
  │
  ▼
创建执行记录（soar_logs）
  │
  ▼
解析 flow_json（节点列表 + 连线）
  │
  ▼
变量解析（全局变量 + 局部变量 + 节点输出引用）
  │
  ▼
初始化所有节点状态为 pending
  │
  ▼
WebSocket 推送执行开始
  │
  ▼
按节点顺序执行
  ├── 普通节点：执行 → 更新状态 → 推送
  ├── 条件分支(P1)：计算条件 → 选择分支
  ├── 循环节点(P1)：遍历列表 → 循环执行
  ├── 并行节点(P1)：并行执行 → 汇聚结果
  ├── 错误处理(P1)：try → catch/finally
  └── 审批流(P1)：等待审批 → 继续/停止
  │
  ▼
记录节点执行结果（soar_logs_detail）
  │
  ▼
更新执行记录状态（soar_logs）
  │
  ▼
WebSocket 推送执行完成
  │
  ▼
触发告警（P1：失败自动告警）
  │
  ▼
生成报告（P2：执行报告）
```

### 2.2 核心类

```python
# core/auto/core.py

class SoarTimer:
    """定时调度器单例"""
    - scheduler: GeventScheduler
    - create_scheduler()
    - start() / shutdown() / pause() / resume()
    - add_date() / add_interval() / add_cron()
    - update_date() / update_interval() / update_cron()
    - remove_job() / get_jobs()

class ManageTimer:
    """远程定时管理服务（RPyC）"""
    - load_config()
    - exposed_remove()
    - exposed_pause()
    - exposed_resume()

def auto_execute(uuid, trigger_type="manual", data=None, ...):
    """剧本执行入口"""
    - 加载剧本配置
    - 校验并发限制
    - 创建执行记录
    - 解析 flow_json
    - 解析变量引用
    - 按顺序执行节点
    - 记录执行结果
    - 更新执行状态
    - 推送 WebSocket 消息
```

### 2.3 变量解析引擎

```python
# 变量引用语法
# ${node-1.result}           - 引用节点输出
# ${node-1.data.user.name}   - 引用嵌套字段
# ${var.global_var_name}     - 引用全局变量
# ${node-1.list[0]}          - 引用数组索引
# ${node-1.name|upper}       - 管道符处理

class VariableResolver:
    """变量解析器"""
    - resolve(expression, node_outputs, variables) -> value
    - validate(flow_json, node_outputs, variables) -> errors
    - parse_field_path(path) -> field_parts
    - apply_pipes(value, pipes) -> transformed_value
```

## 7. 关键设计决策

### 7.1 为什么使用 Flask 而非 Django？

- 当前项目已基于 Flask + Orator 构建，迁移成本高。
- Flask 轻量灵活，适合 API 服务。
- P1 评估迁移到 SQLAlchemy 替代 Orator（Orator 社区活跃度低）。

### 7.2 为什么使用 APScheduler 而非 Celery？

- APScheduler 轻量，无需额外依赖（RabbitMQ/Redis）。
- 当前场景（定时任务 + 间隔调度）APScheduler 已满足。
- P2 分布式场景下评估迁移到 Celery + Redis 作为消息代理。

### 7.3 为什么使用 RPyC 而非 gRPC？

- 当前项目已基于 RPyC 构建。
- RPyC 对 Python 原生支持好，无需定义 proto 文件。
- P2 分布式场景下评估迁移到 gRPC 以提升性能。

### 7.4 为什么前端使用 Vue3 而非 React？

- 当前项目前端已使用 Vue3。
- 与 xdp 项目前端技术栈一致，便于团队共享。
- Element Plus 组件库适合企业级后台应用。
