## 2. RBAC 鉴权架构

### 2.1 鉴权流程

```text
HTTP 请求
  │
  ▼
Gin 路由匹配
  │
  ├── 公开路径（login / healthz / webhook）
  │   └── 直接放行，不经过鉴权
  │
  └── 受保护路径
      │
      ▼
  鉴权中间件 (services/api/middleware/auth.go)
      │
      ├── Step 1: 从 Header 提取 Token
      │   ├── 无 Token → 返回 401 UNAUTHORIZED
      │   └── 有 Token → 继续
      │
      ├── Step 2: 从 Redis 查询 Token → user_id 映射
      │   ├── Token 无效或过期 → 返回 401 UNAUTHORIZED
      │   └── 有效 → 继续
      │
      ├── Step 3: 查询用户角色及权限点
      │   ├── 用户被禁用 → 返回 403 FORBIDDEN
      │   ├── 角色为 super_admin → 跳过权限校验，直接放行
      │   └── 普通角色 → 获取权限点列表，注入 ctx
      │
      ├── Step 4: 校验路由所需权限点
      │   ├── 路由注册时标注所需权限（如 workflow:write）
      │   ├── ctx 中的权限点列表包含该权限 → 放行
      │   └── 不包含 → 返回 403 FORBIDDEN + required_permission
      │
      └── Step 5: 放行 → 调用 Handler
```

### 2.2 权限数据结构

```go
// services/api/middleware/auth.go

// 路由权限标注
type RoutePermission struct {
    Code  string // 权限点编码，如 "workflow:write"
    Scope string // 资源类型，如 "workflow"
}

// 鉴权中间件配置
var ProtectedRoutes = map[string]RoutePermission{
    "POST /api/v1/soar/workflows":       {Code: "workflow:write",   Scope: "workflow"},
    "PUT /api/v1/soar/workflows/:id":    {Code: "workflow:write",   Scope: "workflow"},
    "DELETE /api/v1/soar/workflows/:id": {Code: "workflow:delete",  Scope: "workflow"},
    "POST /api/v1/soar/workflows/:id/execute": {Code: "workflow:execute", Scope: "workflow"},
    "GET /api/v1/soar/users":            {Code: "user:read",        Scope: "user"},
    "POST /api/v1/soar/users":           {Code: "user:manage",      Scope: "user"},
    // ...
}

// 公开路径（不经过鉴权）
var PublicRoutes = []string{
    "POST /api/v1/soar/login",
    "GET /healthz",
    "POST /api/v1/soar/webhook",
}
```

### 2.3 权限点初始化

系统启动时自动将预置的 18 个权限点写入 `soar_permission` 表，并创建 4 个内置角色：

| 角色 | 权限点数量 | 说明 |
|---|---|---|
| `super_admin` | 全部 18 个 | 超级管理员 |
| `workflow_editor` | 10 个 | 剧本编辑者 |
| `workflow_executor` | 5 个 | 仅可执行剧本 |
| `viewer` | 3 个 | 只读用户 |

## 3. 执行引擎架构

### 2.1 执行流程

```text
触发执行（手动/Timer/Webhook/API）
  │
  ▼
校验（剧本是否存在、状态是否可执行、并发限制）
  │
  ▼
创建执行记录（soar_logs）
  │
  ▼
解析 flow_json（节点列表 + 连线）
  │
  ▼
变量解析（全局变量 + 局部变量 + 节点输出引用）
  │
  ▼
初始化所有节点状态为 pending
  │
  ▼
WebSocket 推送执行开始
  │
  ▼
按节点顺序执行
  ├── 普通节点：执行 → 更新状态 → 推送
  ├── 条件分支(P1)：计算条件 → 选择分支
  ├── 循环节点(P1)：遍历列表 → 循环执行
  ├── 并行节点(P1)：并行执行 → 汇聚结果
  ├── 错误处理(P1)：try → catch/finally
  └── 审批流(P1)：等待审批 → 继续/停止
  │
  ▼
记录节点执行结果（soar_logs_detail）
  │
  ▼
更新执行记录状态（soar_logs）
  │
  ▼
WebSocket 推送执行完成
  │
  ▼
触发告警（P1：失败自动告警）
  │
  ▼
生成报告（P2：执行报告）
```

```go
// pkg/storage/redis.go

import "github.com/redis/go-redis/v9"

var RDB *redis.Client

func InitRedis(addr, password string, db int) {
    RDB = redis.NewClient(&redis.Options{
        Addr:     addr,
        Password: password,
        DB:       db,
    })
}

// 设置
RDB.Set(ctx, "token:xxx", "user_id", 24*time.Hour)

// 获取
val, _ := RDB.Get(ctx, "token:xxx").Result()

// 存在性检查
exists, _ := RDB.Exists(ctx, "token:xxx").Result()

// 删除
RDB.Del(ctx, "token:xxx")
```

## 7. 关键设计决策

### 7.1 为什么使用 Go 而非 Python？

- Go 编译为单一二进制文件，部署简单，无需 Python 运行时依赖。
- Go 原生并发模型（goroutine + channel）天然适合执行引擎的并行节点场景。
- Go 静态类型 + 接口编译期检查，减少运行时错误。
- 与 xdp 项目技术栈一致，便于团队复用 Go 基础设施和库。
- Go 生态的 Gin（HTTP）、GORM（ORM）、gorilla/websocket（WebSocket）等库成熟稳定。

### 7.2 为什么使用 Gin 而非其他 Go Web 框架？

- Gin 性能优秀，支持中间件链式处理，适合 RESTful API 服务。
- Gin 路由支持分组，便于按模块组织 API 路径。
- Gin 社区活跃，文档完善，与 xdp 项目技术栈一致。

### 7.3 为什么使用 GORM 而非其他 ORM？

- GORM 是 Go 生态中最流行的 ORM，社区活跃。
- GORM 支持自动迁移、钩子、预加载、软删除等特性。
- 与 xdp 项目技术栈一致。

### 7.4 为什么前端使用 Vue3 而非 React？

- 与 xdp 项目前端技术栈一致，便于团队共享。
- Element Plus 组件库适合企业级后台应用。
- Vue3 Composition API 与 Go 后端结构清晰对应。
