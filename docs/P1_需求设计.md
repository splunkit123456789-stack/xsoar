# SOAR 平台 P1 需求设计

## 1. 阶段目标

P1 面向上线后的产品化增强，重点是把 MVP/P0 中已经跑通的核心能力收敛为可运维、可扩展、可协作的 SOAR 平台能力。

P1 不改变 P0 的核心闭环，而是在剧本编排能力、审批流、审计、版本管理、APP 市场等方面增强。

## 2. P1 主要能力

| 模块 | P1 需求 |
|---|---|
| 剧本审批流 | 高危剧本执行前需审批，审批通过后才执行 |
| 审计日志 | 记录所有用户操作、登录、剧本执行、配置变更 |
| 条件分支节点 | 支持 if/else、switch 等条件分支节点 |
| 循环节点 | 支持对列表/数组进行遍历执行（for/foreach） |
| 并行执行节点 | 支持多个节点并行执行，结果汇聚后继续 |
| 错误处理节点 | 支持 try/catch 风格的错误处理 |
| 剧本版本管理 | 剧本每次保存生成新版本，支持版本对比与回滚 |
| 剧本导入/导出 | 支持剧本以 JSON 文件形式导出与导入 |
| APP 市场/导入 | 支持从外部 ZIP 包导入第三方 APP |
| 节点输出变量提取 | 上游节点的输出可被下游节点通过变量引用 |
| 执行历史与重跑 | 剧本每次执行记录历史，支持基于历史参数重跑 |
| 双主题 UI | 提供亮色/暗色主题切换 |
| APP 参数类型完善 | 支持 text/number/select/textarea/password/checkbox/datetime/file |
| 告警通道 | 内置邮件/钉钉/飞书告警 APP，剧本失败时可触发告警 |

## 3. 剧本审批流 P1 要求

### 3.1 审批配置

- 剧本编辑页可启用"审批流"。
- 启用后需配置审批人（单个或多个）。
- 审批人必须是已注册的 SOAR 用户。
- 审批方式：单人审批（任一审批人通过即可）、会签（所有审批人通过）。

### 3.2 审批流程

1. 剧本被触发（手动/定时/Webhook）。
2. 系统检查剧本是否启用审批流。
3. 启用审批流：剧本进入"待审批"状态，发送通知给审批人。
4. 审批人通过/驳回剧本执行。
5. 通过：剧本执行；驳回：剧本不执行，记录驳回原因。

### 3.3 审批通知

- 通知方式：站内信、邮件、钉钉/飞书（需配置）。
- 通知内容：剧本名称、触发人、触发时间、审批链接。

### 3.4 审批历史

- 审批历史列表展示：剧本名称、触发人、审批人、审批结果、审批时间。
- 审批历史可查询、筛选、导出。

## 4. 审计日志 P1 要求

### 4.1 审计范围

| 操作类型 | 记录内容 |
|---|---|
| 登录/登出 | 用户、时间、IP、User-Agent、成功/失败 |
| 用户管理 | 操作人、目标用户、操作类型（增/改/删） |
| 角色管理 | 操作人、目标角色、操作类型 |
| 剧本管理 | 操作人、目标剧本、操作类型、变更前后摘要 |
| 剧本执行 | 触发人、触发方式、剧本、执行结果 |
| APP 管理 | 操作人、目标 APP、操作类型 |
| 变量管理 | 操作人、目标变量、操作类型 |
| 系统设置 | 操作人、配置项、变更前后值 |

### 4.2 审计日志查询

- 审计日志列表支持按时间范围、用户、操作类型筛选。
- 支持分页（默认 20 条/页，可选 50/100）。
- 支持导出 CSV。

### 4.3 审计日志保留

- 审计日志保留期：180 天。
- 超过保留期的日志自动归档（P1 不实现归档，进入 P2）。
- 审计日志不可删除，仅可查询。

## 5. 剧本编排能力扩展 P1 要求

### 5.1 条件分支节点

- 节点类型：`if`、`else_if`、`else`、`switch`、`case`、`default`。
- 条件表达式支持：`==`、`!=`、`>`、`<`、`>=`、`<=`、`in`、`not in`、`contains`、`matches`（正则）。
- 条件表达式可引用上游节点输出、全局变量、局部变量。
- 编辑器中分支节点以不同颜色区分。
- 分支节点可嵌套。

### 5.2 循环节点

- 节点类型：`for`（固定次数）、`foreach`（遍历列表）、`while`（条件循环）。
- `foreach` 节点需配置：列表来源（上游节点输出或变量）、迭代变量名。
- 循环体内可包含任意节点（包括嵌套循环）。
- 循环最大次数限制：1000 次（防止死循环）。
- 循环执行结果按迭代顺序输出。

### 5.3 并行执行节点

- 节点类型：`parallel`。
- 配置：并行分支数、每分支的子流程。
- 所有分支执行完成后，`parallel` 节点输出汇聚结果。
- 任一分支失败时，`parallel` 节点状态为 `failed`，其他分支继续执行直到完成或超时。
- 并行分支数限制：10 个。

### 5.4 错误处理节点

- 节点类型：`try_catch`。
- 配置：try 分支、catch 分支、可选 finally 分支。
- try 分支中任一节点失败时，跳转到 catch 分支执行。
- catch 分支可访问错误信息 `${error.message}`、`${error.node_id}`。
- finally 分支无论成功失败都执行。

## 6. 剧本版本管理 P1 要求

- 剧本每次保存自动生成新版本，版本号自增（v1, v2, v3...）。
- 剧本详情页展示版本历史列表。
- 支持版本对比：选择两个版本，展示 flow_json 差异（节点增删、参数变更）。
- 支持版本回滚：将剧本回滚到指定历史版本。
- 版本回滚生成新版本（不覆盖历史）。
- 版本最大保留数：50 个，超过自动清理最早版本。
- 当前正在执行的版本不可删除。

## 7. 剧本导入/导出 P1 要求

- 剧本列表页提供"导出"按钮，导出为 JSON 文件。
- 导出 JSON 包含：剧本元信息、flow_json、flow_data、controller_data、local_var_data。
- 导出 JSON 不包含：执行历史、变量值（仅含变量引用）。
- 剧本列表页提供"导入"按钮，选择 JSON 文件导入。
- 导入时若剧本 UUID 已存在，提示用户选择：覆盖、创建新剧本。
- 导入失败时返回明确错误（JSON 格式错误、必填字段缺失等）。

## 8. APP 市场/导入 P1 要求

### 8.1 APP 包格式

- APP 包为 ZIP 文件，解压后包含：
  - `app.json`：APP 元信息（必需）。
  - `main/`：APP 执行入口（Python 模块，必需）。
  - `icon.png`：APP 图标（可选，默认使用系统图标）。
  - `readme.md`：APP 使用说明（可选）。
  - `requirements.txt`：APP 依赖的 Python 包（可选，P1 不自动安装）。

### 8.2 APP 包导入流程

1. 用户在"APP 管理"页面点击"导入 APP"。
2. 选择 ZIP 文件，上传。
3. 系统校验 ZIP 包：
   - 解压成功。
   - `app.json` 存在且格式合法。
   - `main/` 目录存在。
   - `app.json.identification == "soar"`。
4. 校验通过：解压到 `apps/<code>/` 目录，写入 `soar_app` 表。
5. 校验失败：返回明确错误，不写入数据库。

### 8.3 APP 包版本管理

- 同一 APP 导入高版本时覆盖当前版本。
- 同版本再次导入时，提示用户确认覆盖。
- 低版本导入被拒绝，返回 `APP_VERSION_NOT_NEWER`。

### 8.4 APP 启用/禁用

- 禁用的 APP 不可被剧本引用。
- 已被剧本引用的 APP 不可禁用，返回 `APP_IN_USE`。
- 禁用的 APP 在 APP 列表中灰色展示。

## 9. 节点输出变量提取 P1 要求

- 上游节点的输出可被下游节点通过变量引用。
- 引用语法：`${<node_id>.<field_path>}`。
- `field_path` 支持点号访问嵌套字段：`${node1.data.user.name}`。
- 支持数组索引访问：`${node1.list[0]}`。
- 支持管道符处理：`${node1.name|upper}`、`${node1.name|lower}`、`${node1.name|trim}`。
- 变量引用在剧本执行前进行静态校验：
  - 引用的节点是否存在。
  - 引用的字段是否在上游节点输出 schema 中。
  - 引用的全局变量是否存在。

## 10. 执行历史与重跑 P1 要求

- 执行历史列表展示：执行 ID、剧本名称、触发方式、状态、开始时间、耗时。
- 执行历史支持按剧本、时间范围、状态筛选。
- 执行历史详情页展示：每节点状态、输入、输出、错误、耗时。
- 执行历史详情页提供"重跑"按钮：基于历史参数重新执行剧本。
- 重跑生成新的执行 ID，但保留原始执行 ID 的关联（`replay_of` 字段）。

## 11. 双主题 UI P1 要求

- 提供亮色主题（默认）和暗色主题。
- 主题切换入口：用户头像下拉菜单 → 主题切换。
- 主题偏好持久化到用户配置（`soar_user_preference` 表）。
- 主题切换不需要刷新页面。
- 两种主题都符合无障碍标准（WCAG 2.1 AA）。

## 12. APP 参数类型完善 P1 要求

| 参数类型 | 控件 | 校验 |
|---|---|---|
| `text` | 单行输入框 | 最大长度 256 |
| `number` | 数字输入框 | 整数或浮点数 |
| `select` | 下拉选择框 | 值必须在 data 列表中 |
| `textarea` | 多行输入框 | 最大长度 4096 |
| `password` | 密码输入框 | 不回显，存储加密 |
| `checkbox` | 复选框 | 值为 true/false |
| `datetime` | 日期时间选择器 | 格式 YYYY-MM-DD HH:MM:SS |
| `file` | 文件上传 | 最大 10MB，支持类型配置 |

## 13. 告警通道 P1 要求

### 13.1 内置告警 APP

- 邮件告警 APP：复用 `email` APP，预置 SMTP 配置（通过全局变量）。
- 钉钉告警 APP：复用 `dingding` APP，预置 Webhook 配置。
- 飞书告警 APP：复用 `feishu` APP，预置 Webhook 配置。

### 13.2 剧本失败自动告警

- 剧本编辑页可启用"失败自动告警"。
- 启用后需配置告警接收人（邮件地址、钉钉/飞书群 Webhook）。
- 剧本执行失败时，自动触发告警 APP 发送告警。
- 告警内容：剧本名称、执行 ID、失败节点、错误信息、执行时间。

## 14. P1 不包含

- 完整 RBAC（资源级、按钮级权限）。
- 多租户、组织隔离。
- 分布式执行引擎。
- SSO 单点登录。
- 剧本执行报告导出（PDF/HTML）。
- 大屏可视化。
- AI 智能剧本生成。
- 剧本沙箱与安全扫描。
- ChatOps 集成。
- Kubernetes Operator 化。
