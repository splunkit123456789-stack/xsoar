# SOAR 平台 MVP 需求设计

## 1. 阶段目标

MVP 目标是形成一个可部署、可登录、可编排、可执行、可查日志的最小 SOAR 平台闭环。

MVP 不追求完整替代 Splunk SOAR、Demisto、Phantom，而是验证 SOAR 自身从 APP 加载、剧本编排到自动化执行的产品化主路径。

## 2. 核心用户

| 用户 | 诉求 | MVP 高频操作 |
|---|---|---|
| 平台管理员 | 完成基础部署和配置 | 登录、加载 APP、创建剧本、手动执行 |
| 安全运营工程师 | 编排自动化响应剧本 | 拖拽节点、配置参数、执行剧本、查看日志 |
| DevOps 工程师 | 编排运维自动化剧本 | 复用基础 APP，搭建主机/数据库运维流程 |
| 渗透测试工程师 | 编排自动化渗透流程 | 调用 nmap/otx/threatbook 等安全工具 APP |

## 3. 产品信息架构

```text
登录页
  └── 控制台
      ├── 仪表盘（占位，MVP 不要求完整数据）
      ├── 剧本管理
      │   ├── 剧本列表
      │   └── 剧本编辑器（图形化编排）
      ├── APP 管理
      │   ├── APP 列表
      │   └── APP 详情（参数定义）
      └── 执行日志
          ├── 执行历史
          └── 执行详情（节点状态、输出、错误）
```

MVP 控制台只要求核心四个业务页面可用，不要求完整设置中心、报表中心、告警中心或多用户管理。

## 4. MVP 模块范围

| 模块 | MVP 范围 | 不进入 MVP |
|---|---|---|
| 登录 | 单管理员用户名密码登录，返回 Bearer Token | 多用户、RBAC、Token 自助管理、SSO |
| APP 管理 | 扫描 apps/ 目录加载 APP，列表展示，启用/禁用 | APP 在线上传、APP 市场、APP 版本管理 |
| 剧本管理 | 剧本 CRUD、图形化编排、保存 flow_json | 剧本分类、版本管理、剧本审批、剧本导入导出 |
| 剧本执行 | 手动触发执行、节点状态实时推送、执行日志记录 | 定时触发、Webhook 触发、审批流、条件分支、循环节点 |
| 执行日志 | 执行历史列表、执行详情（每节点状态/输出/错误） | 日志导出、日志归档、日志检索 |
| 基础 APP | helloworld、base64、md5、email、http_request 等基础 APP | 复杂安全工具 APP、自定义 APP 开发 |
| 验收 | Docker 一键启动、页面操作、剧本执行验证 | 大规模性能压测 |

## 5. APP 插件化边界

MVP 建立 APP 插件化方向，但只支持内置 APP 加载。

| 类型 | MVP 口径 |
|---|---|
| 内置 APP | 通过 apps/ 目录扫描加载，解析 app.json 元信息 |
| 第三方 APP | 不支持上传，进入 P1 |
| APP 分组 | 不要求分组展示，按字母序列表即可，分组进入 P0 |
| APP 参数类型 | 仅支持 text/number/select/textarea/password，其他类型进入 P1 |
| APP 调试 | 不支持单 APP 调试，必须在剧本中执行，单 APP 调试进入 P2 |

## 6. 剧本编排能力边界

MVP 剧本编排只支持**线性流程**：

| 能力 | MVP 是否支持 |
|---|---|
| 起始节点（start_app） | ✅ 必须 |
| 结束节点（end_app） | ✅ 必须 |
| 中间节点串联 | ✅ 支持 |
| 节点参数配置 | ✅ 支持 |
| 上游输出引用下游输入 | ✅ 支持 `${node.field}` 语法 |
| 条件分支（if/else） | ❌ 进入 P1 |
| 循环节点（for/foreach） | ❌ 进入 P1 |
| 并行节点 | ❌ 进入 P1 |
| 错误处理节点 | ❌ 进入 P1 |
| 子剧本调用 | ❌ 进入 P2 |
| 局部变量 | ✅ 支持 |
| 全局变量引用 | ❌ 进入 P0 |

## 7. 执行引擎 MVP 范围

MVP 执行引擎实现最小化执行能力：

| 能力 | MVP 要求 |
|---|---|
| 执行入口 | 手动触发（点击"执行"按钮） |
| 执行方式 | 同步执行，阻塞当前 HTTP 请求直到完成 |
| 节点执行 | 按 flow_json 中节点顺序依次执行 |
| 状态推送 | 通过 WebSocket 推送节点状态变化（pending/running/success/failed） |
| 日志记录 | 每次执行生成执行记录，每节点记录输出和错误 |
| 失败处理 | 任一节点失败，剧本停止，标记失败原因 |
| 超时控制 | 节点级超时 60s，剧本级超时 600s（硬编码，P1 配置化） |
| 并发控制 | 同一剧本不允许并发执行，返回 `WORKFLOW_ALREADY_RUNNING` |

## 8. MVP 完成标准

- 用户可登录控制台。
- 系统可自动加载 apps/ 目录下的 APP。
- 用户可通过图形化编辑器编排一个线性剧本并保存。
- 用户可手动执行剧本，前端实时看到节点状态变化。
- 执行完成后可在执行日志页查看每节点的输入/输出/错误。
- 端到端链路有人工验收步骤和 Playwright 脚本化验证入口。

## 9. MVP 不做

- 多用户、RBAC、组织隔离。
- 定时触发器、Webhook 触发器（进入 P0）。
- 剧本分类管理（进入 P0）。
- 全局变量管理（进入 P0）。
- 仪表盘统计数据展示（占位即可，完整数据进入 P0）。
- 剧本审批流、审计日志（进入 P1）。
- 条件分支、循环、并行节点（进入 P1）。
- 剧本版本管理、导入导出（进入 P1）。
- 分布式执行（进入 P2）。
- AI 智能剧本生成（进入 P3）。
