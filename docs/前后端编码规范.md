# SOAR 平台 前后端编码规范

本文档用于后续 SOAR 前端、后端、API、数据库和测试开发时引用。若与 `产品需求文档.md` 冲突，以产品需求文档的功能边界和优先级为准。

## 1. 通用原则

### 1.1 代码边界

- 新功能必须先对齐 `产品需求文档.md` 中的模块、优先级和 API 契约，再编码。
- MVP/P0 采用单工作空间模型，不引入完整 RBAC、审批流、插件市场、完整审计等 P1/P2/P3 能力。
- 已明确降级到 P1 的能力不得在 P0 页面或 API 中做半成品入口，除非文档明确要求预置入口。
- 保持模块职责单一：登录只负责认证，APP 只负责加载与元信息，剧本只负责编排与执行，日志只负责记录与查询。
- 不允许把前端临时字段直接透传成后端持久化字段；新增字段必须先进入 API 设计和数据库表结构设计。

### 1.2 命名约束

- Go 包、文件、变量使用简洁英文命名，避免拼音和无意义缩写。
- API JSON 字段统一使用 `snake_case`。
- 数据库字段统一使用 `snake_case`。
- Vue 组件内测试选择器统一使用 `data-testid`，命名使用短横线或模块前缀，如 `nav-workflow`、`input-workflow-name`。
- 内部路由、topic、index、plugin code 等稳定标识只允许小写字母、数字、下划线或短横线，服务端必须做规范化。

### 1.3 错误处理

- 后端错误响应统一使用：

```json
{
  "status": "error",
  "code": "VALIDATION_ERROR",
  "message": "name is required",
  "request_id": "req_..."
}
```

- 前端不展示裸错误堆栈；字段级错误必须靠近字段展示。
- 保存失败不能只提示"保存失败"，必须尽量展示可读字段错误或后端错误码。
- 后端不得吞掉运行时发布、数据库写入、配置校验错误；必须记录日志并返回明确错误。

## 2. 后端 Go 编码规范

### 2.1 工程结构

- `cmd/*` 只放服务入口、环境变量读取、启动流程，不放业务规则。
- `services/api/` 存放 API handler、产品化 API 编排和中间件。
- `pkg/*` 存放可复用领域能力，如 workflow、runtime、plugin、storage、scheduler、variable。
- `plugins/*` 存放 APP 插件实现，插件必须通过统一 metadata、config、execute 能力暴露。
- `migrations/mysql/` 存放数据库迁移脚本。
- 新增 MySQL/Redis 访问能力优先放在 `pkg/storage/*`，不要散落在 handler 中。

### 2.2 Handler 约束

- Handler 负责请求解析、鉴权上下文读取、调用领域逻辑、输出响应。
- Handler 不直接拼复杂 SQL，不承载长流程业务逻辑。
- 每个受保护 API 必须经过统一鉴权中间件，除公开路径外不得绕过 Token 校验。
- API 参数校验必须在进入持久化和运行时发布前完成。
- 对 P0 未支持的能力返回明确错误，例如 `APP_NOT_SUPPORTED`，不要静默保存成不可用配置。

### 2.3 RBAC 鉴权规范

- 所有受保护 API 必须在路由注册时标注所需权限点，格式为 `resource:action`（如 `workflow:read`）。
- 鉴权中间件统一处理 Token 解析 → 用户查询 → 角色查询 → 权限校验。
- 鉴权中间件将权限点列表注入请求上下文，Handler 中无需重复查询数据库。
- 未授权访问统一返回 `403 FORBIDDEN`，响应体包含 `required_permission` 字段。
- 超级管理员角色（`super_admin`）跳过所有权限校验。
- 公开路径（登录、健康检查、Webhook）在路由注册时显式声明，不经过鉴权中间件。
- 菜单权限由前端根据 `GET /api/v1/soar/me/permissions` 返回的权限点列表动态控制。
- 权限点编码采用 `资源:动作` 格式，预置 18 个权限点，系统初始化时写入数据库。

### 2.4 配置与环境变量

- 环境变量读取必须有默认值和说明，新增配置同步更新文档。
- 生产敏感配置不得使用开发默认值，文档必须提示生产部署需修改。
- 配置优先级：环境变量 > 配置文件 > 默认值。
- 本地测试或沙箱环境运行 Go 测试时，使用项目内缓存：

```bash
GOCACHE=/Users/zhulin/xsoar/.cache/go-build \
GOMODCACHE=/Users/zhulin/xsoar/.cache/go-mod \
go test ./...
```

### 2.4 APP 插件开发约束

- APP 必须声明稳定 `plugin_code`、`version`、`plugin_type` 和配置 schema（app.json）。
- APP 必须区分运行时能力与页面渲染能力：`args` 是配置校验权威来源，前端按 `args` 中声明的 `type` 渲染对应控件。
- APP 必须实现 `Execute(args, context) (output, error)` 接口，接收 `args` 和 `context`，返回 `output` 和 `error`。
- APP 不得直接访问数据库，必须通过 `context` 提供的接口。
- APP 不得执行破坏性操作（如删除文件、关闭进程），除非用户在 args 中明确确认。
- APP 必须做参数校验，缺失必填参数时返回 `VALIDATION_ERROR`。
- APP 执行超时由执行引擎控制，APP 内部不得实现自己的超时逻辑。

### 2.5 剧本执行约束

- 剧本执行必须通过执行引擎（`pkg/runtime/`）统一调度，不得在 handler 中直接执行。
- 执行引擎负责节点状态管理、错误处理、超时控制、并发控制。
- 节点执行失败时，剧本停止，标记失败原因。
- 剧本执行必须生成执行日志，记录每节点的输入、输出、状态、耗时。
- WebSocket 推送的节点状态变化必须与数据库中的执行日志一致。

### 2.6 数据库与迁移

- 数据库结构变更必须通过 `migrations/mysql/` 新增迁移文件，不直接修改线上表。
- 表结构设计必须同步到 `数据库表结构设计.md`。
- MySQL 配置表使用软删除字段 `deleted_at`，删除 API 默认逻辑删除。
- JSON 配置字段必须有 schema 说明，不允许长期保存无结构任意 JSON。
- 密码、Token、private key 等敏感字段不得明文落库；P1 使用引用式 `secret_ref` 或加密存储。

## 3. 前端 Vue3 编码规范

### 3.1 工程结构

- 前端工程位于 `web/console`，使用 Vue3 + Vite + Vitest。
- 推荐目录：

```text
web/console/src/
  api/              # API 调用封装
  components/       # 通用组件
  pages/            # 页面组件
  stores/           # Pinia store
  styles/           # 样式文件
  utils/            # 工具函数
  test-utils/       # 测试工具
```

### 3.2 页面与导航约束

- 登录页独立展示，不属于主导航。
- 登录后主导航只展示：仪表盘、剧本管理、APP 管理、执行日志。
- P0 采用顶部唯一主导航，导航项紧跟 `SOAR>Console` 品牌右侧靠左排列。
- 不在 MVP 主导航中加入审批、权限、Dashboard、告警、组织隔离、系统设置等 P1/P2 页面。
- 页面字段必须与需求文档一致，已明确裁剪的字段不得回流到页面。
- MVP 引入 Vue Router，支持登录页 → 控制台 → 各模块页面的路由跳转，URL 可读。
- P0 路由体系包含：登录页、仪表盘、剧本管理列表、剧本编辑器、APP 管理、执行日志、系统设置。
- P1 扩展为完整路由体系，支持 URL 分享、深链直达、权限路由、面包屑、页面级懒加载、404/403 页面和部署 fallback。

### 3.3 视觉主题约束

- P0 登录页必须对齐剧本管理、APP 管理、执行日志的 `ops-console` 浅色业务控制台主题。
- P1 再让剧本管理、APP 管理、执行日志支持与登录页一致的深色品牌主题；P1 允许浅色业务控制台主题和深色品牌主题两套样式并存。
- P0 业务控制台不得沿用登录页橙粉大标题、粉色光晕、营销式 hero 区。
- 业务页主色使用青绿/蓝灰，卡片使用浅灰或白色，按钮使用青绿主按钮。
- 业务控制台根节点必须保留类似 `data-theme="ops-console"` 的主题标识。
- 图标色系按模块区分，但必须保持蓝绿体系，不使用紫色默认 SaaS 风格。

### 3.4 表单与交互

- 必填项必须先做前端拦截，避免明显无效请求打到后端。
- 仍必须保留后端校验，前端校验不能替代后端校验。
- 表单保存前不要提交 UI-only 字段。
- 删除、禁用等破坏性操作必须有明确状态变化和可读反馈。
- 查询、保存、预览等异步操作应有 loading 或 disabled 状态，避免重复提交。

### 3.5 API 调用

- 认证启用后，前端请求受保护 API 必须携带：

```http
token: <login 返回的 token>
```

- P1 评估迁移到标准 `Authorization: Bearer <token>`。
- Token 存储和清理由统一封装处理，业务页面不得各自拼接认证逻辑。
- API 响应解析必须处理非 2xx 错误，并展示 `error.message`。
- 不允许业务页面直接依赖旧字段，例如剧本页不得依赖 `flow_json` 之外的内部字段。

### 3.6 测试选择器

- 关键页面、导航、按钮、输入框必须有稳定 `data-testid`。
- 测试选择器不得依赖展示文案作为唯一定位方式，展示文案可作为断言。
- 新增交互必须补充 Vitest + Vue Test Utils 测试。
- 剧本执行节点状态颜色必须按 `data-node-status` 属性渲染，便于测试断言。

## 4. API 契约规范

### 4.1 基本格式

- REST API 使用 `/api/v1/soar/...`。
- 列表响应统一使用 `data` 数组，含 `total` 和 `pagination`。
- 单对象详情直接返回对象。
- 时间字段使用 `YYYY-MM-DD HH:MM:SS` 格式（北京时间）。
- API 字段使用 `snake_case`，不使用前端 camelCase。

### 4.2 状态码

- `200`：查询、保存、更新成功。
- `400`：请求体非法或字段校验失败。
- `401`：未认证或 Token 无效。
- `403`：已认证但无权限，P0 预留。
- `404`：资源不存在或已删除。
- `409`：唯一键或状态冲突。
- `422`：语义正确但当前阶段不支持，例如 P0 保存 Kafka 运行时配置。
- `429`：超出限流。
- `500`：服务端内部错误。

### 4.3 向后兼容

- 已发布 API 字段不得随意删除；如必须调整，先在文档中标记兼容期。
- 旧字段可以短期兼容读取，但新前端不得继续提交已废弃字段。
- 响应中新增字段必须不破坏旧调用方。

## 5. 数据库规范

### 5.1 表结构

- 主键默认使用 `INT AUTO_INCREMENT`，分布式场景使用 `CHAR(36)` UUID。
- 业务唯一标识使用 `code` 或 `uuid`，并加唯一索引。
- 状态字段统一使用 `TINYINT`（0=启用，1=禁用）或 `VARCHAR(32)`。
- 配置 JSON 字段必须命名为 `*_config`、`*_json` 或具备明确含义。
- 表必须包含 `create_time`、`update_time`；配置类表必须包含 `deleted_at`。

### 5.2 JSON 字段

- JSON 字段必须在需求文档中列出 schema 或字段说明。
- JSON 字段不保存敏感明文。
- JSON 字段里的 key 使用 `snake_case`。

### 5.3 Redis

- Key 命名使用冒号分隔：`token:<token>`、`exec_sum`、`{uuid}&&exec_sum`。
- Token TTL 默认 24 小时。
- 缓存 Key 必须设置 TTL，避免内存泄漏。

## 6. 测试规范

### 6.1 后端测试

- 新增 Go 逻辑必须补 `go test` 覆盖。
- API 行为变更优先补 `services/api/internal/*_test.go`。
- 执行引擎、APP 加载、变量提取等底层能力必须补包级单元测试。
- 沙箱或受限环境统一使用项目内缓存：

```bash
GOCACHE=/Users/zhulin/xsoar/.cache/go-build \
GOMODCACHE=/Users/zhulin/xsoar/.cache/go-mod \
go test ./...
```

- 测试覆盖率目标：核心模块 ≥ 70%。

### 6.2 前端测试

- 前端测试命令：

```bash
cd web/console
npm test        # Vitest 单元测试
npm run build   # 生产构建
```

- 新页面或新交互必须补 `*.test.js`。
- 登录、导航、剧本管理、APP 管理、执行日志五个页面必须保持主流程测试。
- 视觉主题类需求至少通过 DOM 标记和关键 class 做防回归测试。

### 6.3 端到端验证（Playwright）

- 端到端测试使用 Playwright，测试脚本位于 `soar_docs/playwright/tests/`。
- 运行命令：

```bash
cd soar_docs/playwright
npx playwright test
```

- 每个阶段（MVP/P0/P1/P2）必须有对应的 Playwright 测试用例。
- 测试用例必须使用 `data-testid` 选择器，不得依赖展示文案。
- 关键流程（登录、剧本编排、执行、日志查看）必须有端到端覆盖。

### 6.4 端到端验证（一键启动）

- 本地完整体验使用：

```bash
bash scripts/start-oneclick.sh
```

- 真实链路验证优先使用已有脚本：

```bash
bash scripts/real-e2e.sh
bash scripts/verify-mvp.sh
```

## 7. 文档同步规范

- 需求变更同步更新 `产品需求文档.md` 和对应阶段的 `_需求设计.md`。
- 测试口径同步更新 `_人工验收.md` 和 Playwright 测试脚本。
- API 或数据库变更必须同步更新相关章节，不能只改代码。
- 前端视觉、导航、字段裁剪变更必须同步产品文档中的页面要求。
- README 只保留启动、访问、快速验证等用户入口，不承载完整需求细节。

## 8. 禁止事项

- 禁止在剧本页重新展示 `flow_json` 之外的内部字段。
- 禁止前端绕过登录态直接调用受保护 API。
- 禁止后端只做前端校验而缺少服务端校验。
- 禁止把 P1/P2 能力伪装成 P0 已完成。
- 禁止在数据库中明文保存密码、Token、private key。
- 禁止新增接口但不写错误码和响应示例。
- 禁止新增表但不写字段说明和迁移策略。
- 禁止提交无法复现的"手工测试通过"结论，必须写明命令或人工步骤。
- 禁止 APP 直接访问数据库或执行破坏性操作。
- 禁止剧本执行绕过执行引擎直接调用 APP。

## 9. 后续引用方式

后续编码任务开始前，应先引用本规范并确认：

- 当前任务属于哪个模块和优先级。
- 是否需要更新 API 设计或数据库表结构。
- 是否涉及前端页面字段裁剪或视觉主题。
- 需要新增哪些前端、后端、脚本或人工测试用例。
- 哪些能力属于 P1/P2/P3，不应混入本次 MVP/P0 实现。
