# SOAR 平台 P1 人工验收

## 1. 验收目标

验证 P1 的剧本审批流、审计日志、条件分支、循环节点、并行节点、错误处理节点、版本管理、剧本导入导出、APP 市场、节点输出变量提取、执行历史与重跑、双主题 UI、APP 参数类型完善、告警通道真实生效。

## 2. 前置条件

```bash
bash scripts/start-oneclick.sh --clean
```

使用管理员账号登录：

| 项 | 值 |
|---|---|
| 用户名 | `admin` |
| 密码 | `soar_12345678` |

## 3. 剧本审批流验收

### 3.1 启用审批流

1. 创建剧本 `审批测试剧本`，编排 helloworld 节点。
2. 进入剧本详情页，点击"审批流配置"。
3. 启用审批流，选择审批人 `analyst1`，审批方式 `single`，通知渠道 `email`。
4. **通过标准**：保存成功，审批流状态为"已启用"。

### 3.2 触发审批流

1. 使用 `admin` 账号触发剧本执行。
2. **通过标准**：
   - 剧本不立即执行，进入"待审批"状态。
   - 审批人 `analyst1` 收到邮件通知。
   - 审批列表中展示该审批记录。

### 3.3 审批通过

1. 使用 `analyst1` 账号登录，进入"审批管理"。
2. 找到待审批记录，点击"通过"，填写审批意见"确认执行"。
3. **通过标准**：
   - 审批状态变为"已通过"。
   - 剧本自动执行。
   - 执行日志中关联审批记录 ID。

### 3.4 审批驳回

1. 触发另一个审批剧本。
2. 使用 `analyst1` 账号审批驳回，填写驳回原因"剧本有风险"。
3. **通过标准**：
   - 审批状态变为"已驳回"。
   - 剧本不执行。
   - 驳回原因记录在审批历史中。

### 3.5 无权审批验收

1. 使用 `admin` 账号尝试审批由 `analyst1` 审批的记录。
2. **通过标准**：返回错误 `APPROVAL_NO_PERMISSION`。

## 5. 条件分支节点验收

### 5.1 配置条件分支

1. 创建剧本 `条件分支测试`，编排以下节点：
   - 节点 1：helloworld，参数 `name = W5`。
   - 节点 2：if 节点，条件 `${node-1.result} == 'Hello W5'`。
   - 节点 3：if 分支的子节点，helloworld，参数 `name = if-branch`。
   - 节点 4：else 分支的子节点，helloworld，参数 `name = else-branch`。
2. 保存剧本。
3. **通过标准**：编辑器展示条件分支节点，颜色区分。

### 5.2 执行条件分支

1. 执行剧本 `条件分支测试`。
2. **通过标准**：
   - 节点 1 执行成功，输出 `Hello W5`。
   - if 条件成立，执行节点 3，输出 `Hello if-branch`。
   - 节点 4 未执行。

## 6. 循环节点验收

   - 执行日志展示循环迭代详情。

## 7. 并行执行节点验收

### 7.1 配置并行节点

1. 创建剧本 `并行测试`，编排以下节点：
   - 节点 1：parallel 节点，配置 3 个并行分支。
   - 分支 1：helloworld，参数 `name = branch1`，sleep 2s。
   - 分支 2：helloworld，参数 `name = branch2`，sleep 3s。
   - 分支 3：helloworld，参数 `name = branch3`，sleep 1s。
2. 保存剧本。
3. **通过标准**：编辑器展示并行节点，3 个分支并行排列。

### 7.2 执行并行

1. 执行剧本 `并行测试`。
2. **通过标准**：
   - 3 个分支并行执行。
   - 总执行时间 ≈ 3s（最慢分支时间），而非 6s（串行时间）。
   - parallel 节点输出汇聚 3 个分支的结果。

## 8. 错误处理节点验收

3. **通过标准**：保存成功，变量引用通过静态校验。

### 12.2 执行变量引用

2. **通过标准**：
   - 节点 1 输出 `Hello W5`。

## 16. 告警通道验收

### 16.1 启用失败告警

1. 创建剧本 `告警测试`，编排 helloworld 节点，参数 `name = ${invalid_var}`（故意制造错误）。
2. 进入剧本详情页，点击"失败告警配置"。
3. 启用失败告警，配置告警渠道：
   - email：`admin@example.com`
   - dingding：`https://oapi.dingtalk.com/robot/send?access_token=xxx`
4. **通过标准**：保存成功，失败告警状态为"已启用"。

### 16.2 触发失败告警

1. 执行剧本 `告警测试`。
2. **通过标准**：
   - 剧本执行失败。
   - 系统自动触发 email APP 发送告警邮件到 `admin@example.com`。
   - 系统自动触发 dingding APP 发送告警消息到配置的钉钉群。
   - 告警内容包含：剧本名称、执行 ID、失败节点、错误信息、执行时间。
